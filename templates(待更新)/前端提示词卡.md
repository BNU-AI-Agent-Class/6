# 前端提示词卡（BUILD/B3 已完成版）

## 0. B3 结论

**阶段状态：B3 · 前端界面已完成。**

当前前端页面已经放入：

```text
frontend/
└── index.html
```

它是一个纯静态单页聊天界面，无框架、无构建工具、无模型 key，只调用后端 `POST /api/chat`。页面包含真正隐藏式侧边栏、常驻危机求助按钮、历史对话、设置面板、按情绪分组的微信式 `+` 表情入口、头像昵称、AI“正在思考”状态和断后端 fallback。正式界面不展示后端状态或模型 key 状态。

## 1. 三句话主提示词

| 句 | 内容 |
|---|---|
| 1. 做什么页 | 做一个单页聊天界面，历史消息在上，输入框在下。用户发送后，将前端维护的 `messages` 数组 POST 到后端 `/api/chat`，把后端返回的 `reply` 显示成 AI 气泡。 |
| 2. 什么风格 | 贴合“留学生情绪梳理 Agent”的陪伴式定位：温暖、安静、圆角、低压力，不做落地页，首屏就是可直接对话的工具界面。 |
| 3. 连哪个后端 | 后端地址用变量 `const BACKEND = "http://localhost:8000"`；上线后在 B4/SHIP 阶段改成部署后的后端域名。 |

## 2. 追加要求

```text
- 前端只能调用自己的后端 /api/chat 和 /api/health，不能出现任何模型 API key，不能 fetch OpenRouter/OpenAI/DeepSeek。
- 页面常驻“我需要紧急帮助”按钮，点击后发送“我现在很难受，撑不下去了，需要紧急帮助。”，触发后端危机拦截。
- 后端连不上时不要白屏：普通消息显示中性兜底；只有前端检测到明确本人危机意图或用户点击紧急帮助时才显示真人求助资源。
- 前端不传 system prompt，system prompt 由后端自动注入 agent.md + skills。
- 根据后端返回的 is_crisis、safety_light、stage 调整气泡样式和阶段提示。
- 侧边栏可以折叠；设置里提供个性化、存储、记忆三类配置。
- 输入框左侧加 `+` 按钮，打开表情面板。
- AI 头像下显示用户自定义的智能体名字；用户头像下显示用户名。
```

## 3. 当前页面功能

| 功能 | 实现 |
|---|---|
| 聊天主界面 | `#messages` 显示用户/AI 气泡，输入框在底部 |
| 气泡对齐 | 气泡顶部与头像顶部对齐，头像下方显示名字 |
| 头像配色 | AI 与用户头像统一使用温暖的 Claude 黄 `#d7a06d`，文字使用深棕色 |
| 折叠侧边栏 | 收起后侧栏宽度归零，只保留左上角菜单图标；展开后显示产品介绍、历史记录与设置 |
| 发送消息 | `POST ${BACKEND}/api/chat`，请求体为 `{ messages: history }` |
| 危机求助按钮 | 常驻侧栏按钮，点击发送危机求助文本 |
| 快捷开场 | 三个低压力开场按钮：室友没打招呼、融不进去、想试小行动 |
| 微信式输入 | 输入框左侧 `+` 打开表情面板；按平静、难过、焦虑、生气、孤独、疲惫、被理解、有希望分组，每组 3 个表情 |
| 回答状态 | 请求发出后立即显示 AI 头像、名称和“正在思考”动画，收到结果后原位移除并展示回复 |
| 优雅降级 | 普通断线显示中性 fallback；仅当本地检测到明确危机表达时展示危机资源，不白屏、不把网络故障染红 |
| 危机/橙灯提示 | 根据 `is_crisis`、`safety_light` 显示红色危机气泡或系统提示 |
| 历史对话 | 使用 `localStorage` 保存本地历史，侧边栏可切换 |
| 删除对话 | 历史列表每条对话带垃圾桶按钮，删除后进入备份列表 |
| 设置-个性化 | 可改字号、从多种中英文字体中选择字体，并修改 AI 名字、用户名字；不开放背景色，统一使用暖灰白低对比主题 |
| 设置-存储 | 导出当前对话、查看本地缓存地址、查看本地回收站、永久清空回收站、永久清除全部本地对话与设置 |
| 设置-记忆 | 可开启/关闭对话之间记忆，并填写用户自我描述 |

## 4. 与 B2 API 契约的对应关系

### 请求

```javascript
await fetch(BACKEND + "/api/chat", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ messages: buildRequestMessages(conversation) })
});
```

### 响应使用字段

| 字段 | 前端用途 |
|---|---|
| `reply` | 显示为 AI 气泡 |
| `is_crisis` | 判断是否追加真人求助提示 |
| `safety_light` | `red` 显示危机样式，`orange` 显示有害环境提示 |
| `stage` | 更新顶部“阶段” |
| `model` | B3 暂不直接展示；B4 可用于调试面板 |

开启“对话之间记忆”时，前端会在本次请求的 `messages` 开头加入一条“用户自我描述”背景消息；关闭时不注入。

## 5. 安全与降级

- **零 key：** `frontend/index.html` 不包含任何 API key，也不读取环境变量。
- **零直连模型：** 前端只 fetch `http://localhost:8000/api/health` 和 `http://localhost:8000/api/chat`。
- **危机入口：** 用户随时可以点击“我需要紧急帮助”。
- **断后端兜底：** 普通网络失败只展示中性 fallback，不得标红；明确危机意图才展示真人求助资源。
- **本地缓存：** 历史对话、个性化设置、记忆开关和自我描述存于浏览器 `localStorage` 的 `bnu-agent-ui-v5`。发送给后端时只保留 `user/assistant`，过滤 `system` 和旧危机 AI 回复。
- **版本握手：** 每次发送前检查 `/api/health` 的版本为 `3.4.0`；旧后端不会收到消息，前端提示重启服务。
- **不替代真人帮助：** 页面底部固定提示：若有伤害自己或无法保证安全的感觉，优先联系真人或当地紧急服务。

## 6. 登录页判断

当前不建议在 B3 做真正登录功能。原因是登录需要后端认证、用户表、密码存储/哈希、会话或 token、安全退出等配套设计；如果现在只做一个假登录页，答辩时容易被问到“数据到底保护在哪里”。更合适的顺序是：

1. B4 先完成前后端连通与安全自检。
2. SHIP 前如果需要展示入口，可以做一个“前端静态登录样式页/占位页”。
3. 真正账号系统留到后端认证设计完成后再做。
