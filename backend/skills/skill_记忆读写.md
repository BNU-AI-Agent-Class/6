# skill_记忆读写（T5 v3.4）

## 目标

让对话在用户可控、可删除的前提下保持必要的连续性，并准确描述当前实现。当前版本没有账号系统、服务端用户数据库、跨设备同步、订阅保存或“三个月危机历史”。

## 当前实现的三类上下文

### 1. 当前会话历史

前端把当前对话中的 `user` / `assistant` 消息保存在当前浏览器的 `localStorage` 中。发送新消息时，前端把当前会话历史随 `POST /api/chat` 发送给后端；后端不建立用户档案，也不把对话写入数据库。

安全判定只统计本次请求中当前会话已经出现的用户安全信号，不把它描述成跨会话或按日历计算的观察窗。

### 2. 用户可选的自我描述

用户可以在“设置—记忆”中主动开启记忆并填写自我描述。开启后，这段文字由前端加入每次请求的消息历史，用来提供背景。关闭后不再发送；用户可以修改或通过“永久清除全部本地对话与设置”删除。

不得从对话中自动推断并写入诊断、真实身份、学校、住址、联系方式或其他敏感画像。

### 3. 产品级语义与程序知识

多元视角、低压力行动、安全边界和陪伴流程来自 `agent.md` 与技能文件，是所有用户共用的产品规则，不是为某个用户建立的长期记忆。

## 读取规则

1. 只读取本次 API 请求实际携带的当前会话历史。
2. 仅当用户开启记忆且填写自我描述时，使用该段背景。
3. 不声称知道未随请求传入的旧对话，也不声称拥有跨设备记忆。
4. 不把 AI 过去生成的危机回复当成新的用户风险信号。
5. 对当前最新用户消息的安全判断优先于旧历史。

## 写入与删除规则

1. 对话和设置只由前端写入当前浏览器的 `localStorage`。
2. 普通“删除对话”先移入本地回收站，避免误删；回收站内容仍然保存在浏览器中。
3. 用户可永久清空回收站，也可永久清除全部本地对话与设置；永久删除不可撤销。
4. 后端当前不提供 `memory_note`、用户档案写入或服务端删除接口，因为后端不持久化这些数据。
5. 不承诺“永不删除”“订阅制保存”“最近三次完整对话 + 早期摘要”等尚未实现的能力。

## 数据流与隐私边界

- 每次发送时，所选对话历史会经过本项目后端并交给后端配置的模型服务商生成回复。
- 不使用“端到端加密”描述当前数据流；HTTPS 保护传输，但后端与模型服务商需要处理请求内容。
- 不要求真实姓名或可识别信息，并在界面提醒用户不要输入敏感个人信息。
- 本地删除只影响当前浏览器，不能撤回已发送并由服务商按其政策处理的既往请求。
- 不将本地会话自动用于产品改进，也不向无关第三方分享。

## 对话措辞约束

可以说：

> 这段对话保存在你当前使用的浏览器中；如果开启“记忆”，你填写的自我描述会随请求一起发送。你可以在设置里永久清除本地数据。

不能说：

> 我会永远记得你。

> 所有记录都只在本地、任何服务端都看不到。

> 我已经替你保存了三个月的危机记录。
